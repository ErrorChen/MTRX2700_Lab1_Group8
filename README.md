# MTRX2700 Lab 1 Group 8

## 1. Project Summary

This repository contains a GNU Arm Assembly implementation of the USYD MTRX2700 STM32 assembly lab for the STM32F3 Discovery board (`STM32F303VCTx`, Cortex-M4).

The active codebase is modular and keeps `Src/platform_defs.inc` as the single source of truth for:

- compile-time exercise selection
- UART and timer tunables
- register addresses and bit masks
- framing constants
- checksum mode selection

The baseline lab path still supports Exercises 1 to 5 with the original BCC/XOR checksum framing. A bonus extension adds a software CRC16 path that can be enabled at compile time without changing the exercise entry flow.


## 5. Compile-Time Configuration

Edit `Src/platform_defs.inc` before building.

Important selectors:

- `ACTIVE_EXERCISE`
  - `1` to `5`
- `EX2_ACTIVE_MODE`
  - `EX2_MODE_BUTTON`
  - `EX2_MODE_TIMED`
- `EX5_ACTIVE_ROLE`
  - `EX5_ROLE_TX`
  - `EX5_ROLE_RX`
- `CHECKSUM_ACTIVE_MODE`
  - `CHECKSUM_MODE_BCC`
  - `CHECKSUM_MODE_CRC16`

Recommended baseline setting:

```asm
.equ CHECKSUM_ACTIVE_MODE,               CHECKSUM_MODE_BCC
```

Recommended bonus setting:

```asm
.equ CHECKSUM_ACTIVE_MODE,               CHECKSUM_MODE_CRC16
```

## 6. Framing Protocol

### Common rules

- Byte 0 is `STX` (`0x02`)
- Byte 1 is `LEN`
- `LEN` is the total frame length in bytes, from `STX` through the final checksum byte
- Payload begins at byte offset 2
- `ETX` (`0x03`) is always placed immediately before the checksum field
- ACK and NAK use the same framing logic as normal payload messages

If payload length is `N`:

- BCC frame length is `N + 4`
- CRC16 frame length is `N + 5`

### Baseline BCC mode

Format:

```text
[STX][LEN][PAYLOAD...][ETX][BCC]
```

`BCC` is the original 8-bit XOR checksum across all previous frame bytes.

### Bonus CRC16 mode

Format:

```text
[STX][LEN][PAYLOAD...][ETX][CRC_H][CRC_L]
```

CRC variant:

- name: `CRC-16/CCITT-FALSE`
- polynomial: `0x1021`
- initial value: `0xFFFF`
- reflect input: `false`
- reflect output: `false`
- final XOR: `0x0000`
- appended byte order: high byte first, then low byte
- standard check value for `"123456789"`: `0x29B1`

The CRC is implemented in software in `Src/crc16.s`. This was chosen instead of the STM32 hardware CRC peripheral so the bonus stays transparent, easy to inspect, and self-contained in assembly.

## 7. Baseline Lab Functionality

The baseline marking path is preserved:

- Exercise 1 still demonstrates string operations, frame generation, and checksum verification in RAM.
- Exercise 2 still implements the LED counter with button or timed stepping.
- Exercise 3 still sends on UART4, receives on UART5, validates the frame, and replies with ACK or NAK.
- Exercise 4 still validates TIM2 timing and LED blink scheduling.
- Exercise 5 still implements the TX/RX counter protocol with ACK/NAK handling.

Switching to CRC16 does not change exercise entry points, payload format, or role selection. It only changes the checksum field width and verification logic inside the shared framing layer.

## 8. Bonus Extension Summary

The bonus path adds:

- a compile-time checksum selector in `Src/platform_defs.inc`
- derived frame-length constants so BCC and CRC16 share the same framing owner
- `Src/crc16.s` with software CRC16 calculation and verification
- generic checksum dispatch in `Src/frame_codec.s`
- UART receive and ACK/NAK logic that follow the active checksum mode automatically
- Exercise 1 RAM symbols for a deterministic CRC16 self-test and active-mode visibility

Bonus-specific Exercise 1 evidence:

- `exercise1ActiveChecksumMode`
- `exercise1ActiveChecksumBytes`
- `exercise1ChecksumValid`
- `exercise1CorruptChecksumValid`
- `exercise1Crc16SelfTestExpected`
- `exercise1Crc16SelfTestActual`
- `exercise1Crc16SelfTestPass`

## 9. Build and Run in STM32CubeIDE

Recommended workflow:

1. Open or import the root project in STM32CubeIDE.
2. Edit `Src/platform_defs.inc` to select the exercise, role, and checksum mode.
3. Run `Project -> Clean`.
4. Run `Project -> Build Project`.
5. Flash and debug the board.

Why `Clean` is recommended:

- it forces CubeIDE to regenerate the managed build files
- it guarantees the new `Src/crc16.s` module is picked up cleanly

## 10. UART Wiring Assumptions

Board pin mapping:

- UART4: `PC10` TX, `PC11` RX
- UART5: `PC12` TX, `PD2` RX
- alternate function: `AF5`

Exercise 3 wiring:

- connect `PC10` to `PD2` to route the transmitted UART4 frame into UART5 receive
- if you want to observe the generated ACK/NAK reply, monitor `PC12` with a logic analyzer, USB-UART adapter, or second board

Exercise 5 wiring with two boards:

- Board A `PC10` -> Board B `PC11`
- Board B `PC10` -> Board A `PC11`
- common ground between boards is required
- use 3.3 V UART signalling only

## 11. Testing Procedure

### Exercise 1

Set:

- `ACTIVE_EXERCISE = 1`

Check in the debugger:

- string lengths are populated correctly
- `exercise1FrameLength` matches the selected checksum mode
- `exercise1ChecksumValid == 1`
- `exercise1CorruptChecksumValid == 0`
- `exercise1Crc16SelfTestActual == 0x29B1`
- `exercise1Crc16SelfTestPass == 1`

### Exercise 3

Set:

- `ACTIVE_EXERCISE = 3`

Procedure:

- press the USER button
- verify a framed payload is sent on UART4
- verify UART5 receives and validates the frame
- inspect `exercise3LastStatus`
- observe ACK LED pattern on success or NAK LED pattern on failure

This works in both BCC and CRC16 modes because `uartReceiveFramedMessage` now uses the active checksum verifier.

### Exercise 5

Set:

- `ACTIVE_EXERCISE = 5`
- `EX5_ACTIVE_ROLE = EX5_ROLE_TX` on one board
- `EX5_ACTIVE_ROLE = EX5_ROLE_RX` on the other board

Procedure:

- verify TX sends `COUNTER = XXX` frames periodically
- verify RX accepts valid frames, updates LEDs, and returns ACK
- verify TX increments the counter after ACK
- verify RX sends NAK and flashes LEDs if the frame or payload is invalid
- verify TX resets the counter and flashes LEDs on NAK or timeout

## 12. Suggested Tutor Demo

### Baseline compatibility demo

1. Set `CHECKSUM_ACTIVE_MODE = CHECKSUM_MODE_BCC`.
2. Run Exercise 1 and show `exercise1ChecksumValid == 1`.
3. Run Exercise 5 on two boards and show the original counter protocol still works.

### Bonus demo

1. Change only `CHECKSUM_ACTIVE_MODE` to `CHECKSUM_MODE_CRC16`.
2. Rebuild.
3. Run Exercise 1 and show:
   - `exercise1ActiveChecksumMode == CHECKSUM_MODE_CRC16`
   - `exercise1ActiveChecksumBytes == 2`
   - `exercise1ChecksumValid == 1`
   - `exercise1CorruptChecksumValid == 0`
   - `exercise1Crc16SelfTestActual == 0x29B1`
   - `exercise1Crc16SelfTestPass == 1`
4. Run Exercise 5 on two boards and show valid CRC16 frames are still accepted end to end.

This gives tutors both:

- a deterministic local proof that a corrupted CRC16 frame is rejected
- an integration demo showing the same protocol still works over UART with the stronger checksum

## 13. Assumptions and Limitations

- Checksum mode is compile-time selected. Both ends of a UART link must be built with the same mode.
- The protocol still uses an 8-bit `LEN` field. The configured payload and frame sizes remain well within that limit.
- UART, timer, and button services are polling-based and blocking.
- The bonus path improves checksum robustness, but it does not add runtime mode negotiation or retransmission logic beyond the existing ACK/NAK behaviour.
- The README documents the active root project. Legacy snapshot folders are intentionally not described as active modules.
